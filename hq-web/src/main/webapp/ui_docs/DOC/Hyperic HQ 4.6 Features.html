Product Plugin Management and Deployment

Hyperic 4.6 provides new features that ease the process of deploying and managing product plugins. These features, described in the paragraphs below, include a user interface for managing product plugins, and new Hyperic Server and Hyperic Agent behaviors for ensuring consistent plugin deployment across a Hyperic environment.

Plugin Manager

Plugin Manager is a new user interface that enables you to deploy and manage plugins across a Hyperic deployment from the Hyperic user interface. You can use Plugin Manager to deploy and remove plugins on the Hyperic Server and all agents reporting to the server.

Server-Agent Plugin Synchronization

The new Server-Agent Plugin Synchronization (SAPS) feature ensures that product plugins are consistently deployed across a Hyperic deployment. SAPS is enabled by new behaviors on the Hyperic Server and Hyperic Agent. When a Hyperic 4.6 Agent starts up, it sends a plugin status report to the Hyperic Server. If an agent's deployed plugins do not exactly match those deployed on the Hyperic server, the server updates the agent's plugins appropriately. Similarly, each time the Hyperic Server is restarted, it compares the product plugins it deploys to most recent plugin status reported by each agent — if an agent's deployed plugins do not exactly match those deployed on the Hyperic server, the server updates the agent's plugins appropriately. SAPS is enabled by default; it can be disabled using the new server.pluginsync.enable server property, described below.

New Hyperic Server Property to Enable and Disable SAPS

Hyperic 4.6 provides a new Hyperic Server property, server.pluginsync.enable, which you can use to disable the SAPS feature described in the previous section. To disable SAPS, add server.pluginsync.enable=false to ServerHome/conf/hq-server.conf, and restart the Hyperic Server.

New Plugin Descriptor Element to Support SAPS

As of Hyperic 4.6, all product plugins must have a version. A new attribute — version — is required in the root <plugin> element in the XML descriptor for a product plugin.

iPad and iPhone Support

The Hyperic 4.6 user interface can be access from iPads and iPhones.

Updated Plugins

All Hyperic plugins' XML descriptors have been updated to define the version of the plugin as a <property> element named "version".

In addition, the following Hyperic plugins are updated with new functionality, or to support more recent versions of the managed product:

  • GemFire
  • RabbitMQ — In Hyperic 4.6, the RabbitMQ plugin no long requires access the erlang.cookie file — the plugin now uses the rabbitmq-management plugin. The configuration process has changed, and additional metrics are available. For more information see RabbitMQ.
  • tc Server — Support for tc Server 7.0 added.
  • Jboss — Support for Jboss 5.1 and 6.0 added.
  • PostgreSQL — Support for PostgreSQL 9.0 added.
  • WebSphere
  • WebLogic Server Plugin — In Hyperic 4.6, the plugin for managing WebLogic Server is enhanced to enable discovery of node managers and managed servers in a WebLogic domain without accessing the Administration Server for the domain. In previous releases, it was necessary to discover the Administration Server for a domain, and then configure that resource with WebLogic Server credentials, in order to discover the node manager and managed servers in the domain.

HQApi Updates

Hyperic 4.6 includes HQApi 5.0. Changes include:

  • client.properties — In HQApi 5.0, client.properties supports encrypted passwords, and can be stored in either ~/.hq or hqapi1-client-n.n.n/conf. In previous versions, the APIs look for server connection properties in ~/.hq/client.properties; to use a connection properties in a different location you had to supply the path to the client.properties on the command line, using the --properties command qualifier. (Supplying the properties file location on the command line is still supported.) For more information about specifing connection properties in a file, see HQApi Command-Line Tools
  • groovyshell — This new API allow you to run Groovy scripts from the command shell.
  • maintenance — This API is updated to enable maintenance downtime to be scheduled for individual resources. In previous releases, maintenance could only be scheduled for a group of resources.
